# 修复收藏数计算逻辑 - 产品需求文档

## Overview
- **Summary**: 修复视频收藏数的计算逻辑，确保收藏数只跟用户是否有收藏这个视频有关，一个用户把一个视频收藏再多收藏夹都只计算为一次，避免出现负数的收藏数。
- **Purpose**: 解决收藏数计算错误的问题，确保收藏数的准确性和合理性。
- **Target Users**: 所有使用视频收藏功能的用户。

## Goals
- 修复收藏数计算逻辑，确保一个用户对一个视频的收藏只计算一次。
- 确保收藏数不会出现负数。
- 保持收藏功能的其他功能正常运行。

## Non-Goals (Out of Scope)
- 不改变收藏夹的其他功能。
- 不修改收藏夹的创建、编辑、删除功能。
- 不修改视频的其他互动功能（如点赞、投币、分享等）。

## Background & Context
- 当前的收藏数计算逻辑是：每当视频被添加到一个收藏夹时，收藏数增加 1；每当视频从一个收藏夹中移除时，收藏数减少 1。
- 这种计算方式导致一个用户把一个视频收藏到多个收藏夹时，收藏数会被多次计算。
- 当用户从多个收藏夹中移除同一个视频时，收藏数可能会被多次减少，导致出现负数。

## Functional Requirements
- **FR-1**: 收藏数应该只跟用户是否有收藏这个视频有关，一个用户对一个视频的收藏只计算一次。
- **FR-2**: 收藏数应该始终是非负数。
- **FR-3**: 当用户第一次收藏视频时，收藏数应该增加 1。
- **FR-4**: 当用户取消所有收藏夹中的视频时，收藏数应该减少 1。
- **FR-5**: 当用户在不同收藏夹之间移动视频时，收藏数应该保持不变。

## Non-Functional Requirements
- **NFR-1**: 修复后的收藏数计算逻辑应该与现有的收藏夹功能兼容。
- **NFR-2**: 修复后的收藏数计算逻辑应该具有良好的性能，不会影响系统的响应速度。

## Constraints
- **Technical**: 修复应该在现有的代码架构基础上进行，不应该引入新的依赖。
- **Dependencies**: 依赖于现有的收藏夹和视频数据模型。

## Assumptions
- 每个用户对每个视频的收藏状态是独立的，与收藏夹的数量无关。
- 收藏数的计算应该基于用户对视频的收藏状态，而不是基于收藏夹的数量。

## Acceptance Criteria

### AC-1: 收藏数计算正确
- **Given**: 用户将一个视频收藏到多个收藏夹
- **When**: 系统计算视频的收藏数
- **Then**: 收藏数只增加 1
- **Verification**: `programmatic`
- **Notes**: 无论用户将视频收藏到多少个收藏夹，收藏数都只增加 1。

### AC-2: 收藏数不会出现负数
- **Given**: 用户将一个视频从所有收藏夹中移除
- **When**: 系统计算视频的收藏数
- **Then**: 收藏数减少 1，但不会小于 0
- **Verification**: `programmatic`
- **Notes**: 即使用户从多个收藏夹中移除同一个视频，收藏数也只会减少 1，并且不会出现负数。

### AC-3: 收藏数在收藏夹之间移动时保持不变
- **Given**: 用户将一个视频从一个收藏夹移动到另一个收藏夹
- **When**: 系统计算视频的收藏数
- **Then**: 收藏数保持不变
- **Verification**: `programmatic`
- **Notes**: 当用户在不同收藏夹之间移动视频时，收藏数应该保持不变。
