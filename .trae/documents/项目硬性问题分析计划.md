# 项目硬性问题分析计划

## 一、安全配置问题（高风险）

### 问题1：数据库密码硬编码
- **位置**：`mybilibili-admin/src/main/resources/application.yml` 第15行
- **问题**：`password: ${MYSQL_PASSWORD:}` 明文硬编码在配置文件中
- **风险**：如果代码提交到版本库，数据库密码将泄露
- **建议**：使用环境变量或配置中心管理敏感配置

### 问题2：DeepSeek API Key 硬编码
- **位置**：`mybilibili-admin/src/main/resources/application.yml` 第79行
- **问题**：`api-key: "sk-7ab7abb58465a9e7032ed942f77468477424896392dd3af06b39ab9c49503a09"` 明文硬编码
- **风险**：API Key 泄露可能导致被他人在你的账号下消耗配额
- **建议**：使用环境变量 `AI_DEEPSEEK_API_KEY` 替代

### 问题3：管理员账号密码硬编码
- **位置**：`mybilibili-admin/src/main/resources/application.yml` 第89-90行
- **问题**：`admin: admin` / `password: admin123` 明文硬编码
- **风险**：生产环境中使用默认密码极不安全
- **建议**：强制要求首次部署时修改默认密码，或使用加密配置

---

## 二、日志输出问题（中风险）

### 问题4：大量使用 System.out.println
- **位置**：16个文件发现 `System.out.println` 使用
- **主要文件**：
  - `VideoServiceImpl.java` - 多处调试输出
  - `VideoProcessServiceImpl.java` - 视频处理流程日志
  - `AdminSubtitleServiceImpl.java` - 字幕服务日志
  - `AiSummaryController.java` - AI功能日志
- **风险**：
  - 生产环境无法通过日志级别控制
  - 无法写入日志文件
  - 性能影响（同步输出到stdout）
- **建议**：统一使用 SLF4J/Logback 框架

### 问题5：异常使用 e.printStackTrace()
- **位置**：多个文件的异常处理中
- **风险**：
  - 输出到 stderr，难以追踪
  - 生产环境无法集中收集
  - 可能造成内存溢出（大量异常输出）
- **建议**：使用 `log.error("message", e)` 或 `log.debug("message", e)`

---

## 三、资源泄漏风险（中风险）

### 问题6：文件流未使用 try-with-resources
- **位置**：
  - `UserServiceImpl.java`
  - `VideoProcessServiceImpl.java`
  - `SubtitleTextUtils.java`
  - `AiSummaryServiceImpl.java`
  - `FileUploadUtils.java`
- **问题**：直接使用 `new FileInputStream`/`new FileOutputStream`/`new FileReader`/`new FileWriter`
- **风险**：如果发生异常，文件流不会被正确关闭，导致资源泄漏
- **建议**：使用 `try-with-resources` 语句确保资源关闭

---

## 四、代码规范问题（低风险）

### 问题7：SQL字符串拼接方式
- **位置**：
  - `ManuscriptMapper.java` - 多处 `+` 拼接字符串构建SQL
  - `VideoMapper.java`
  - `CommentMapper.java`
- **说明**：虽然当前使用 MyBatis 注解 + `<script>` 标签，参数化查询是安全的，但字符串拼接可读性差
- **建议**：使用 StringBuilder 或统一工具类构建动态SQL

## 修复步骤

### 第一阶段：安全配置修复（P0）
1. 修改 `application.yml`，将数据库密码、API Key、管理员密码改为环境变量引用
2. 创建 `.env.example` 文件作为环境变量配置模板

### 第二阶段：日志框架改造（P1）
1. 替换所有 `System.out.println` 为 `log.info/debug`
2. 替换所有 `e.printStackTrace()` 为 `log.error`
3. 确保使用了 `@Slf4j` 或 `Logger` 声明

### 第三阶段：资源管理优化（P2）
1. 将所有文件流操作改为 `try-with-resources`
2. 添加资源关闭的异常处理

### 第四阶段：代码规范优化（P3）
1. 重构 SQL 构建逻辑，使用 StringBuilder 或工具类
2. 统一代码风格
