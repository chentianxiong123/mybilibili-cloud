# 前后端联调接口标准文档

## 1. 文档概述

本文档旨在规范前后端联调过程中的接口标准，确保前端与后端之间的交互一致性，提高开发效率，减少沟通成本。文档基于当前项目的数据库结构和业务需求，定义了统一的接口规范、数据格式和交互流程。

## 2. 数据库结构与实体映射

### 2.1 核心数据表

| 表名 | 描述 | 对应实体类 |
|------|------|------------|
| `users` | 用户信息表 | `User` |
| `manuscripts` | 稿件表 | `Manuscript` |
| `videos` | 视频表 | `Video` |
| `comments` | 评论表 | `Comment` |
| `user_interactions` | 用户互动表 | `UserInteraction` |
| `messages` | 消息表 | `Message` |
| `conversations` | 会话表 | `Conversation` |
| `message_settings` | 消息设置表 | `MessageSetting` |
| `notifications` | 通知表 | `Notification` |
| `coins` | 投币表 | `Coin` |
| `shares` | 分享表 | `Share` |
| `favorite_folders` | 收藏夹表 | `FavoriteFolder` |
| `favorite_videos` | 收藏视频表 | `ManuscriptFavorite` |

### 2.2 实体关系

- **User** 与 **Manuscript**：一对多（一个用户可以创建多个稿件）
- **Manuscript** 与 **Video**：一对多（一个稿件可以包含多个视频分P）
- **User** 与 **Comment**：一对多（一个用户可以发表多条评论）
- **Manuscript** 与 **Comment**：一对多（一个稿件可以有多条评论）
- **User** 与 **UserInteraction**：一对多（一个用户可以有多次互动）
- **User** 与 **Message**：一对多（一个用户可以发送/接收多条消息）
- **User** 与 **FavoriteFolder**：一对多（一个用户可以创建多个收藏夹）
- **FavoriteFolder** 与 **Manuscript**：多对多（通过favorite_videos表关联）

## 3. API接口规范

### 3.1 接口命名规范

1. **路径规范**：
   - 统一使用小写字母
   - 单词间使用连字符（-）分隔
   - 资源路径使用复数形式
   - 示例：`/api/manuscripts`, `/api/users/123`

2. **方法规范**：
   - `GET`：获取资源
   - `POST`：创建资源
   - `PUT`：更新资源
   - `DELETE`：删除资源

3. **参数规范**：
   - 路径参数：用于唯一标识资源，如 `/api/manuscripts/{id}`
   - 查询参数：用于过滤、排序、分页等，如 `/api/manuscripts?page=1&size=10`
   - 请求体：用于传递复杂数据，如创建或更新资源时

### 3.2 统一响应格式

所有API接口应返回统一的响应格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

- `code`：状态码，200表示成功，其他表示失败
- `message`：响应消息，成功时为"操作成功"，失败时为具体错误信息
- `data`：响应数据，根据接口返回具体数据结构

### 3.3 错误处理规范

| 状态码 | 描述 | 错误信息示例 |
|--------|------|--------------|
| 400 | 请求参数错误 | "请求参数格式错误" |
| 401 | 未授权 | "未登录或登录已过期" |
| 403 | 禁止访问 | "没有权限访问该资源" |
| 404 | 资源不存在 | "请求的资源不存在" |
| 500 | 服务器内部错误 | "服务器内部错误，请稍后重试" |

## 5. 前端API调用规范

### 5.1 API封装

前端应统一封装API调用，使用axios创建实例，并添加请求和响应拦截器：

```javascript
// 创建axios实例
const api = axios.create({
  baseURL: '/api',
  timeout: 30000,
  headers: {
    'Content-Type': 'application/json'
  },
  withCredentials: true
});

// 请求拦截器
api.interceptors.request.use(
  config => {
    const token = localStorage.getItem('token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  error => {
    return Promise.reject(error);
  }
);

// 响应拦截器
api.interceptors.response.use(
  response => {
    return response.data;
  },
  error => {
    if (error.response) {
      switch (error.response.status) {
        case 401:
          localStorage.removeItem('token');
          localStorage.removeItem('user');
          break;
        case 403:
          ElMessage.error('没有权限访问该资源');
          break;
        case 404:
          ElMessage.error('请求的资源不存在');
          break;
        case 500:
          ElMessage.error('服务器内部错误');
          break;
        default:
          ElMessage.error(error.response.data?.message || '请求失败');
      }
    } else {
      ElMessage.error('网络错误，请检查网络连接');
    }
    return Promise.reject(error);
  }
);
```

### 5.3 错误处理

前端应统一处理API错误，包括：

1. 网络错误：提示网络连接问题
2. 401错误：清除token，跳转到登录页
3. 403错误：提示权限不足
4. 404错误：提示资源不存在
5. 500错误：提示服务器内部错误
6. 业务错误：根据返回的错误信息提示用户

## 6. 安全规范

### 6.1 认证与授权

1. **JWT认证**：使用Bearer token进行认证
2. **权限控制**：后端实现基于角色的权限控制
3. **防CSRF攻击**：使用CSRF token
4. **防XSS攻击**：对用户输入进行过滤和转义

### 6.2 数据安全

1. **密码加密**：使用bcrypt等算法加密存储密码
2. **敏感数据保护**：不在前端存储敏感数据
3. **HTTPS传输**：使用HTTPS协议传输数据

## 7. 性能优化

### 7.1 前端优化

1. **缓存策略**：使用localStorage或sessionStorage缓存频繁访问的数据
2. **懒加载**：对图片、视频等资源使用懒加载
3. **防抖和节流**：对频繁触发的事件（如搜索、滚动）使用防抖和节流
4. **批量请求**：合并多个API请求，减少网络请求次数

### 7.2 后端优化

1. **数据库索引**：为常用查询字段创建索引
2. **缓存**：使用Redis缓存热点数据
3. **分页查询**：对列表数据使用分页查询
4. **异步处理**：对耗时操作（如视频转码）使用异步处理

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 401错误 | 未登录或token过期 | 重新登录获取新token |
| 403错误 | 没有权限访问该资源 | 检查用户权限，或联系管理员 |
| 404错误 | 接口路径错误或资源不存在 | 检查接口路径是否正确，或确认资源是否存在 |
| 500错误 | 服务器内部错误 | 查看服务器日志，定位错误原因 |
| 网络错误 | 网络连接问题 | 检查网络连接，或稍后重试 |
| 数据格式错误 | 请求参数格式不正确 | 检查请求参数格式是否符合接口要求 |
